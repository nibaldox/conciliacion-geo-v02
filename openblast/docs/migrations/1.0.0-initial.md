# 1.0.0 — Initial Release

This is the **first stable release** of the OpenBlast CSV format.

## What's included

### Format
- **CSV with one row per loaded blasthole.** Plain RFC-4180 CSV.
- **17 mandatory fields** across 6 semantic blocks (Identification, Location, Geometry, Charge, Stemming, Metadata).
- **17 optional fields** in a 7th block (rock properties, advanced survey, comments, anomaly flags).
- **`additionalProperties: true`** at the item level — vendors can include proprietary columns that pass through the schema unchanged.

### Schema
- `openblast/schema/openblast-1.0.0.schema.json` — JSON Schema (Draft-07). Validated row-by-row with `jsonschema` Python or any compliant validator.
- `openblast/schema/openblast-1.0.0.table-schema.json` — Frictionless Data Table Schema equivalent, for tools that consume that format.

### Examples
- `openblast/examples/minimal.csv` — 5 wells, only mandatory fields. Smallest possible valid file.
- `openblast/examples/complete.csv` — 5 wells, all fields populated. Use as a template.
- `openblast/examples/datapackage.json` — Frictionless Data Package descriptor that wraps `complete.csv` with CRS, license, and contributor metadata.

### Vendor mappings (machine-readable)
- `openblast/mappings/enaex-to-openblast.csv`
- `openblast/mappings/datamine-to-openblast.csv`
- `openblast/mappings/surpac-to-openblast.csv`

These CSVs have six columns: `openblast_field`, `<vendor>_field`, `<vendor>_table` (when applicable), `<vendor>_dtype`, `transformation`, `notes`. They document the conversion logic but are **not** executable code — for runtime conversion, use the `openblast-convert` CLI (planned for v1.1).

## What's NOT included (planned for future versions)

| Feature | Target version | Rationale |
|---|---|---|
| `openblast-validate` CLI (Python) | v1.1 | Tooling, not the format itself. |
| `openblast-convert` CLI (Python) | v1.1 | Implementation of the vendor mappings as code. |
| `fragmentation_p50_mm`, `fragmentation_p80_mm` | v1.1 (optional) | Post-blast image analysis (photogrammetry / Split Engineering). |
| `vibration_ppv_mm_s`, `vibration_freq_hz` | v1.1 (optional) | Seismograph data from blast monitoring. |
| Multi-table support (split collar/survey/charge) | v1.2 (optional) | Required by Datamine-style pipelines. |
| Geologic model fields (lithology, alteration) | v2.0 (mandatory) | Cross-domain expansion. |

## Breaking changes planned for 2.0.0

- `shot_at` will switch from ISO-8601 to Unix epoch milliseconds (easier to consume programmatically).
- `initiation_pattern` will become a mandatory field.
- `comment` will be split into `comment_internal` and `comment_external` for clarity.

These will only land after v1.x has been stable for at least 12 months.

## Upgrade policy

There is no previous version, so no upgrade is required. If you were using the legacy `conciliacion-geo-v02` ENAEX-only Excel upload, you should now:

1. Export your ENAEX Excel to OpenBlast CSV using the mapping in `openblast/mappings/enaex-to-openblast.csv` (or wait for v1.1's `openblast-convert` CLI to do it automatically).
2. Validate with the JSON Schema.
3. Upload the resulting CSV.
